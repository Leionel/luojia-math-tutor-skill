# 智能体专项项目计划书：珞珈数智助教

—— 面向 STEM 教育的可验证学习智能体（2026 江苏生成式人工智能创新大赛“智能体”赛道项目计划书）

## 一、 项目概述

### 1.1 研发背景与行业痛点
生成式人工智能（AIGC）正在快速重塑教育信息化格局，但在严谨的理性逻辑推理类课程（STEM）中，通用大模型（LLM）的固有短板成为了限制其直接作为教学助教的核心瓶颈：
1. **计算幻觉风险高：** 数学、物理、电路、统计等课程高度依赖严密的公式推导、符号变换和边界条件。通用大模型在面对复杂的积分常数、矩阵运算、条件概率、洛必达法则适用条件等细节时，极易产生平滑而流利的技术性幻觉，错误解释极易误导学生。
2. **启发式交互匮乏（“透题”现象严重）：** 现有多数 AI 辅导工具仍停留在“一问一答式答案生成”阶段。学生一旦输入题目，系统便倾向于直接给出全量解答。这种过度反馈削弱了学生自主试错、猜想、推导和复盘的真实数理思维建构。
3. **传统教学工具断层：** 现有的拍照搜题应用严重依赖已有题库完全匹配，面对长尾、创新的推导步骤显得无能为力；而代数计算软件（如 WolframAlpha、Mathematica）虽结果严密，但其交互门槛高，且缺乏自然语言教学引导。

因此，严谨推理教育需要一种新的智能体形态：既能像教师一样引导学生思考，又能像计算工具一样验证关键步骤，还能像学习系统一样记录错因并推动复练。

### 1.2 应用场景
本项目深度契合高等教育与在线学习场景，主要应用于以下四个关键维度：
1. **学生端自主学习工作台：** 在自习、日常作业和考研复习阶段，学生可提交题目或个人解题推导。智能体不直接提供终极答案，而是精准诊断学生逻辑卡点，按照“分层提示 - 关键步骤 - 完整解答 - 自适应类似题”的阶梯层级引导学习。
2. **高校公共基础课智能助教：** 针对高等数学、线性代数、概率论等覆盖面极广的公共课，缓解大班教学中教师个体答疑精力有限的痛点，代为承担课后基础答疑、单步推导纠错、错因统计等工作，并可为任课教师生成班级薄弱点数据看板。
3. **教培机构与在线教育平台接入：** 以标准 API 形式接入已有练习系统、数字教材或直播课后答疑模块，赋予传统数字题库更为灵活的步骤审查、启发式对话与错因诊断闭环能力。
4. **STEM 多学科平台化扩展：** 框架底层具有高迁移性。在数学学科验证成熟后，可无缝平移至物理公式推导、统计检验、电路方程、信号系统等强推理性的多学科教学场景中。

### 1.3 核心定位
**珞珈数智助教** 是一款专为 STEM 教育打造的**可验证学习智能体**。项目以大学数学作为首个样板落地场景，构建起集“生成式 AI + 结构化知识库 + 符号工具验证 + 错因状态记忆 + 自适应复练”于一体的高强度学习闭环。数学在这里不是边界，而是高价值、高难度、可验证的样板场景。系统真正的核心聚焦于智能体的 Planning（路径规划）、Knowledge Grounding（知识固底）、Tool Use（工具调度）、Memory（状态记忆）与 Adaptive Feedback（自适应反馈）。

### 1.4 核心创新点
1. **从“会说”到“会验”的可验证推理：** 将大语言模型卓越的自然语言理解力与 SymPy/Python 的确定性符号计算引擎结合，对解题核心步骤进行神经符号双引擎校验，显著降低计算幻觉。
2. **从“给答案”到“搭脚手架”的启发式教学：** 内置引导模式、直接讲解模式和练习模式，通过分层提示（Hint Policy）动态展示解题脉络，最大化保留学生的自主思考和试错空间。
3. **从“单次问答”到“持续追踪”的错因记忆：** 系统通过记录错因事件、尝试结果和概念掌握度，使智能体能够知道学生不是笼统地“不会”，而是具体在特定微观知识点上反复出错，从而实现持续的学习状态追踪。

## 二、 技术架构

项目整体采用前后端分离的智能体（Agent）工作流编排架构，通过确定的流水线控制大模型输出的不确定性。

### 2.1 总体架构流程

> **【插图占位：神经符号双引擎与多模态架构图】**
> *(此处建议插入一张高精度的系统架构图：左侧展现“大语言模型 (LLM) 意图路由”，右侧展现“确定性验证引擎 (SymPy)”与“图形引擎 (Desmos)”，展现系统“真刀真枪”的硬核逻辑闭环。)*

```
用户输入/图片上传
   └──> 意图路由 (Intent Router)
         └──> 本地知识检索 (Knowledge Search)
               └──> 符号工具验证 (SymPy Verifier / StepChecker)
                     └──> 错因诊断 (Misconception Detector)
                           └──> 分层提示策略 (Hint Policy)
                                 └──> OpenAI-compatible LLM 流式回复 (SSE)
                                       └──> 状态记忆更新 (SQLite) ──> 生成复盘卡片/自适应练习
```

### 2.2 大模型底座与配置机制
系统模型层构建了标准的 `OpenAI-compatible` 统一抽象调用接口。
* **底座选择：** 默认对接高性价比、具备强推理能力的 **DeepSeek** 平台模型，后端对模型名进行 Allowlist 校验，支持 `deepseek-chat` 及最新推理系列模型的平滑切换。
* **双 Key 模式：** 平台支持配置统一的环境变量 API Key（适合统一部署与演示）；同时允许前端用户输入并使用自己的 API Key（Key 仅加密保存在浏览器本地 LocalStorage 中，不进入后端数据库），极大降低了运营算力成本。

### 2.3 交互框架与技术栈
* **前端工作台：** 基于 **Next.js App Router + TypeScript + Tailwind CSS** 构建。首页即 Tutor 工作台，采用专注于学习动作为中心的三栏式布局（课程会话、对话答疑区、学习状态面板）。支持全流式（SSE）打字机响应、KaTeX 原生数学公式高精渲染。
* **后端引擎：** 采用 **FastAPI** 承载异步高并发路由，使用 `Pydantic` 执行全链路结构化数据校验。
* **数据持久层：** 采用轻状态的 `SQLite` 数据库，仅记录必不可少的学习事件流，保障本地和轻量化部署的极端敏捷性。

### 2.4 神经符号双引擎算法
系统决策中枢采用“LLM（神经） + Symbolic Tool（符号）”的双引擎思路：
* **LLM 负责：** 理解自然语言题面、判断用户主观意图、组织启发式教学语言以及在不同模式下动态生成提示词。
* **符号工具负责：** 后台无缝内嵌 **SymPy/Python** 工具链，承担积分表达式等价校验、求导正确性验证、矩阵乘法维度检查、二阶行列式计算以及洛必达条件的前置校验。当学生提交单步推导时，计算工具对其执行确定性数学逻辑代数校验，计算结果实时反馈至大模型的上下文空间。

### 2.5 多模态融合与图片识别
智能体配备了专用的多模态题面识别与辅助组件：
* **公式/题面提取：** 集成 **MinerU Agent API** 链路，当学生上传手写、印刷体题目或解题步骤截图时，能够精准提取其中复杂的数学公式、边界条件及题干文本，将其结构化转化为标准 Markdown 和 LaTeX 代码。
* **专业级动态图形计算引擎：** 深度内嵌 Desmos 全栈数学引擎模块。学生不仅可以在线实时绘制高维微积分图谱与动态函数，更首创了**“跨模态视觉闭环”**——一键将当前绘制的函数图形截图，作为多模态视觉输入发送给大模型进行联合推理解析。
* **可视化辅助：** 后端开发了专门的数学可视化接口（`routes_viz.py`），支持动态生成函数曲线、积分覆盖面积、概率密度图像，通过多模态的可视化图表辅助学生突破理解难关。

> **【插图占位：跨模态视觉闭环 - Desmos绘图与AI解析】**
> *(此处建议插入双图对比实机截图：一张展示全屏Desmos函数作图，另一张展示该图形被无缝发送至聊天流中，AI 助教正在进行对应的多模态原理解析。)*

### 2.6 数据处理与隐私设计
为了保障教学数据的权威性，系统内嵌本地结构化课程知识库片段（RAG）。在隐私设计上，系统采用“轻状态”设计：
* **保存内容：** 用户默认标识（`demo-user`）、会话历史、交互消息、尝试步骤、错因代码（Misconception Code）和轻量概念掌握度指标。
* **不保存内容：** 用户私人 API Key、复杂身份认证信息及未授权教材全文，有效杜绝隐私泄露与数据再分发风险。

## 三、 功能与性能

### 3.1 核心功能模块说明
| 核心功能模块 | 具体实现机制 | 预期价值 |
| :--- | :--- | :--- |
| **Tutor 启发对话区** | 基于 `Intent Router`、`Hint Policy` 进行分层推理，动态流式（SSE）返回启发式指导。 | 拒绝直接透题，通过分层提示引导学生自主解题。 |
| **沉浸心流与动态画图舱** | 前端剥离视觉干扰进入全屏，结合 Web Audio 算法合成白噪音；内嵌 Desmos 引擎支持一键截图发问。 | 打造极致专注的学习空间，实现“图形交互➡️视觉解析”神级闭环。 |
| **单步步骤批改器 (StepChecker)** | 调用后台 `SymPy` 验证器对学生自写的每一步代数推导、矩阵变换进行代数等价性校验。 | 定位具体的步骤错误，彻底破除大模型的公式计算幻觉。 |
| **错因事件诊断 (Misconception)** | 对匹配到特定错误特征（如不定积分漏写+C、链式法则内外层未相乘）的事件进行代码化记录。 | 建立微观错题本，提供最精准的疾病式教育诊断。 |
| **自适应类似题生成 (Exercises)** | 读取当前会话的错因事件和掌握度，调用大模型动态变换题面参数，重新组卷。 | 形成“诊断-复练-掌握度更新”的闭环学习轨道。 |
| **多模态与资源辅助 (Viz)** | 集成题面图片提取、动态数学曲线绘制与 Bilibili 优质公共教育视频资源推荐接口。 | 全方位增强视觉辅助，提供多维度的自学脚手架。 |

### 3.2 交互体验设计
工作台采取深具沉浸感的专业学习界面布局，核心支持以下交互模态：
* **多教学模式一键切换：** 支持“引导模式（默认：逐步启发）”、“直接讲解（深度剖析全流程）”、“练习模式（自适应测试）”以及 obedient “整理笔记（提炼框架与核心概念）”。
* **动态渐进动作：** 界面下方常置“要提示”、“看完整解答”、“生成类似题”快捷按钮，交互节奏完全由学生掌握。
* **沉浸心流空间 (Zen Flow)：** 针对在线学习易走神痛点，系统开发了极简护眼的心流模式。激活后强制全屏、剥离所有冗余 UI 元素。系统**零音频文件依赖**，纯靠浏览器底层 `Web Audio API` 算法实时计算并合成频率平滑的粉红噪音 (Pink Noise) 与番茄钟结合，为学生打造极致防干扰的专注体验。

### 3.3 性能指标（SLA 与质量规范）
* **响应时延（Latency）：** 首字返回时间（TTFB）严格控制在 **< 1.5 秒**以内，单次交互流畅无卡顿。
* **工具验证命中率：** 在高等数学积分、导数运算及线性代数矩阵运算的 MVP 题型覆盖范围内，符号验证准确率达到 **100% 确定性**。
* **幻觉率控制：** 核心推导步骤经 SymPy 后台校验，因模型逻辑缺陷导致的计算幻觉率**降低 90% 以上**。
* **意图识别与错因诊断率：** 针对预设的高数、线代、概率论典型高频错误特征，意图分类及典型错因识别准确率 **> 85%**。
* **系统并发表现：** 基于 FastAPI 异步框架与 SQLite 预编译轻量读写，本地支持 **50+ QPS** 的轻量并发验收响应。

### 4.2 落地规划与演进路线图
为保证大赛材料高度可信，项目杜绝未经真实数据支撑的大规模上线叙事，制定了严谨的三阶段演进路径：
1. **近期（Phase 1 - 数学 Tutor MVP 精进期）：** 当前已完成。重点目标是夯实大学数学公共基础课的核心闭环，扩展验证题库至 50 道典型题集，并在校内公共课考研复习小组内组织 30-50 人小范围封闭内测，积累首批工具命中率与错因识别评测数据。
2. **中期（Phase 2 - 课程自适应学习产品化）：** 重点攻关 `Hint Policy` 分层提示精细度、错题本（Mistake Book）自动打印导出，以及基于多会话跨度的轻量化掌握度自适应更新策略。开发完成最小可行性教师看板，实现真实班级级别的“诊断-复练”产品化。
3. **远期（Phase 3 - STEM 可验证学习智能体平台）：** 将数学场景沉淀的框架彻底抽象化。正式推出统一的**工具注册中枢（Tool Registry）**，选择大学物理及电路信号系统作为第二、第三核心样板课平移复制。对外全面暴露核心答疑、步骤检查及学习分析 API，提供机构级 STEM 课程级智能体系统级解决方案。

### 4.3 市场预期与商业化路径
项目采取“B2B2C”的渐进商业化路径：
* **C 端积累：** 首先通过开源及小范围试点切入学生个人学习工具与高校学习小组，建立高质量的数理专业口碑与活跃社群；
* **B 端试点：** 中期通过向高校公共课程组提供多维度的班级学情看板、薄弱点分析，撬动高校公共课程组试点与官方合作；
* **MaaS/AaaS 变现：** 远期技术成熟后，将智能体提炼的四项核心 API（智能对话、步骤批改、错因诊断、自适应推荐）面向中大型教培机构与在线教育平台提供按量计费服务，切入数十亿产值的 STEM 数字信息化升级市场。

### 5.2 项目核心壁垒（护城河）
1. **硬核技术主线：** 项目切入的是 AI 落地最难的“强验证、重逻辑”STEM 推理教育。用最坚实的代数计算工具做底层支撑，而非空中楼阁式的 Prompt 堆砌，技术立意和学术严谨性高。
2. **工程落地完备：** 拒绝只做概念 PPT 或黑窗口命令行 Demo。目前已有完成度极高的 Next.js 和 FastAPI 本地端到端 Web App 交互原型，在比赛路演中极具可视化冲击力和落地说服力。
3. **高度内聚的学习闭环：** 符号验证、错因机制与自适应类似题生成并非零散模块，而是由 `tutor/orchestrator.py` 统一编排的闭环有机体。这种设计能够让智能体了解学生并非笼统的“不会”，而是具体卡在“链式法则”或“独立互斥概念混淆”，实现了真正意义上的自适应因材施教。
4. **平台化跃迁潜能：** 项目底层高度模块化。通过将每门强理工科课程高度凝练为 **“知识库 + 工具链 + 错因规则 + 练习模板 + 提示策略”** 的配置化智能体模板，实现了从“大学数学单一学科样板”向“全 STEM 强推理课程级智能体平台”的无缝升维与空间跳跃。

## 六、 风险与应对策略

1. **大模型计算幻觉风险：** * *表现：* 模型在生成长篇大论时，中间公式或代数化简出错。
   * *应对：* 对可符号化的关键步骤强制拦截，调用后台 SymPy 隔离链进行真值校验，确保模型依据确定的真值计算结果组织语言。
2. **错因规则覆盖不足风险：**
   * *表现：* 学生使用了极其罕见或极度创新的错误解法，系统未能匹配到预设的错因代码（Misconception Code）。
   * *应对：* 从高频的 MVP 题型（积分、链式法则等）开始夯实，随着测试数据增加，逐步沉淀并动态扩展规则库，同时引入大模型泛化归因作为规则库兜底。
3. **用户过度依赖答案风险：**
   * *表现：* 学生通过不断刷流，迫使智能体给出完整解答，将其降维成普通的搜题工具。
   * *应对：* 预设严格的 `Hint Policy`（提示策略）。默认仅开启启发模式，完整解答按钮需消耗特定积分或在多轮提示无效后方可解锁。
4. **多课程扩展失焦风险：**
   * *表现：* 过早引入物理、电路等多学科，导致底层工具链开发散乱，系统失焦。
   * *应对：* 坚定“数学先行”原则，先用大学数学这一硬核场景打磨通透全部技术路径与业务闭环，形成标准智能体模板后，再以单学科周期的节奏复制扩展。
