# 珞珈数智助教：面向 STEM 教育的可验证学习智能体

—— 2026 江苏生成式人工智能创新大赛“智能体”赛道项目计划书草案

## 0. 摘要

珞珈数智助教是一个面向 STEM 教育的可验证学习智能体，以大学数学为首个落地场景，构建“生成式 AI + 结构化知识库 + 符号工具验证 + 错因记忆 + 自适应练习”的学习闭环。

项目当前已完成本地 Web App 原型：前端基于 Next.js，后端基于 FastAPI，支持高等数学、线性代数、概率论基础题型的对话答疑、知识库检索、SymPy 验算、SSE 流式输出、SQLite 轻状态记忆、错因事件记录、掌握度更新、类似题生成、图片上传识别、可视化辅助和视频资源推荐。模型层采用 OpenAI-compatible 抽象，默认对接 DeepSeek，并支持平台 Key 与用户自带 Key 两种模式。

本项目的核心价值不是“让 AI 多讲几道题”，而是用大学数学这一高强度、强验证场景，验证一种可迁移到更多 STEM 课程的智能体范式：智能体能理解学生输入、规划教学路径、调用确定性工具检查关键步骤、诊断错误原因、调整提示深度，并持续形成复练任务。

## 1. 项目概述

### 1.1 项目背景

生成式人工智能正在快速进入教育领域，但在严谨推理类课程中，通用大模型仍存在明显短板。

第一，数学、物理、电路、统计等 STEM 课程高度依赖公式推导、符号变换和条件判断。通用大模型生成的文本往往很流畅，但在积分常数、矩阵运算、条件概率、洛必达适用条件等细节上容易出现计算幻觉。一旦错误解释被学生接受，后续学习会被持续误导。

第二，很多 AI 辅导产品仍停留在“问答式答案生成”。学生一提问，系统就直接给出完整解法。这种方式短期看似高效，但容易让学生跳过试错、猜想、验证和复盘过程，削弱真实的数学能力建构。

第三，传统搜题工具和科学计算软件各有局限。搜题工具依赖题库匹配，难以处理学生自己的推导步骤；科学计算软件结果可靠，但交互门槛高，缺少自然语言教学引导；通用聊天机器人易用，但缺少工具验证和学习状态追踪。

因此，严谨推理教育需要一种新的智能体形态：既能像教师一样引导学生思考，又能像计算工具一样验证关键步骤，还能像学习系统一样记录错因并推动复练。

### 1.2 项目目标

项目目标是打造一个可验证、可记忆、可复练、可迁移的 STEM 学习智能体。

当前阶段目标：

- 以大学数学公共基础课为首个场景，覆盖高数、线代、概率论中的高频基础题型。
- 构建“学生输入 -> 意图识别 -> 知识检索 -> 工具验证 -> 错因诊断 -> Tutor 回复 -> 复盘与复练”的闭环。
- 通过本地开发版 Web App 展示完整产品交互，而不是只展示 Prompt 或命令行 Demo。
- 在比赛材料中形成“数学落地、STEM 延伸、教师端与 API 扩展”的清晰路线。

长期目标：

- 将大学数学中验证过的知识库、工具链、错因库和教学策略抽象为课程级智能体模板。
- 逐步扩展到大学物理、统计与数据分析、电路与信号系统等强推理课程。
- 面向高校公共基础课、在线教育平台和教培机构提供智能答疑、步骤诊断、错因分析和学习数据看板能力。

### 1.3 应用场景

**学生端自主学习**

学生在自习、作业和考研复习中提交题目或自己的推导步骤。系统不急于直接给最终答案，而是判断学生卡在哪个知识点，并按“提示、关键步骤、完整解答、类似题”的层级提供支持。

**高校公共基础课助教**

高数、线代、概率论等公共课学生数量大、答疑需求高、教师精力有限。智能体可以承担课后基础答疑、步骤批改、错因收集与复练推荐，辅助教师发现班级高频薄弱点。

**教培机构与在线教育平台**

教培机构可将“可验证 Tutor Agent”接入题库、练习系统或直播课后答疑场景，获得比普通聊天机器人更可靠的步骤检查和错因诊断能力。

**STEM 课程平台化扩展**

数学是首个验证场景。未来同一套架构可迁移到物理公式推导、统计检验、电路方程、信号系统等课程，形成可配置的课程级智能体平台。

### 1.4 项目定位

推荐比赛定位：

> 珞珈数智助教是面向 STEM 教育的可验证学习智能体，以大学数学为首个落地场景，构建“生成式 AI + 符号验证 + 错因记忆 + 自适应练习”的学习闭环。

这一定避免把项目讲成单薄的“数学教学 App”。数学在这里不是边界，而是高价值、高难度、可验证的样板场景。真正的核心能力是：

- **Planning**：识别学科、题型、意图和教学模式。
- **Knowledge Grounding**：检索本地结构化知识库，提供课程依据。
- **Tool Use**：调用 SymPy/Python 等确定性工具验证关键步骤。
- **Memory**：保存会话、消息、尝试、错因事件和轻量掌握度。
- **Adaptive Feedback**：根据错因、掌握度和用户模式调整提示层级。
- **Practice Loop**：生成类似题与复盘卡片，推动学生继续练习。

## 2. 技术方案

### 2.1 总体架构

项目采用前后端分离的智能体工作流架构。

```text
用户输入/图片上传
  -> Intent Router
  -> Knowledge Search
  -> Step Checker / Verifier
  -> Misconception Detector
  -> Hint Policy
  -> Prompt Builder
  -> OpenAI-compatible LLM Stream
  -> Memory Update
  -> Review Card / Similar Exercise / Learning Panel
```

系统由五层组成：

| 层级 | 作用 | 当前实现 |
|---|---|---|
| 交互层 | Tutor 工作台、流式对话、公式渲染、模式切换 | Next.js App Router、Tailwind、KaTeX、SSE |
| 编排层 | 意图识别、学科判断、提示策略、回复组织 | Intent Router、Orchestrator、Prompt Builder、Hint Policy |
| 知识层 | 本地课程知识依据与关键词检索 | `luojia-math-tutor/references/output/*.json` |
| 工具层 | 对关键推导进行确定性验证 | SymPy Verifier、StepChecker、可视化接口 |
| 记忆层 | 轻状态学习记录 | SQLite 会话、消息、尝试、错因、掌握度 |

### 2.2 前端工作台

前端采用 Next.js App Router + TypeScript + Tailwind 构建，首页即 Tutor 工作台。界面围绕真实学习动作组织，而不是营销式 landing page。

当前核心能力：

- 三栏式学习界面：课程/会话、对话区、学习状态面板。
- 类 Claude/GPT 流式聊天体验。
- KaTeX 公式渲染。
- `Enter` 发送，`Shift+Enter` 换行。
- 支持“引导模式 / 直接讲解 / 练习模式 / 整理笔记”等交互模式。
- 支持“要提示 / 看完整解答 / 生成类似题”等学习动作。
- 用户可选择平台默认 Key 或本地保存自己的 API Key。
- 模型下拉支持 DeepSeek 相关模型配置。
- 图片上传可触发公式/题面提取。
- 错题本、掌握度、视频资源和复盘信息可在界面中呈现。

### 2.3 后端智能体编排

后端采用 FastAPI + Pydantic + SQLite + SymPy。核心模块包括：

- `routes_sessions.py`：会话创建、历史消息读取、会话维护。
- `routes_tutor.py`：Tutor SSE 流式接口。
- `routes_models.py`：模型连通性测试。
- `routes_exercises.py`：类似题生成。
- `routes_mistakes.py`：错因记录与错题本。
- `routes_mastery.py`：轻量掌握度更新与查询。
- `routes_viz.py`：数学可视化接口。
- `routes_uploads.py`：图片上传与题面/公式提取。
- `knowledge/search.py`：本地知识库关键词检索。
- `math_tools/verifier.py`：SymPy 验证。
- `math_tools/step_checker.py`：学生单步推导检查。
- `tutor/misconception.py`：错因规则识别。
- `tutor/orchestrator.py`：串联检索、验证、提示词和 LLM 流式回复。
- `llm/openai_compatible.py`：统一模型调用接口。

### 2.4 神经符号双引擎

项目采用“LLM + Symbolic Tool”的神经符号双引擎思路。

LLM 负责：

- 理解自然语言题目。
- 判断用户意图。
- 组织教学语言。
- 在不同教学模式下生成合适提示。

SymPy/Python 工具负责：

- 验证积分候选答案。
- 验证求导结果。
- 检查表达式等价。
- 计算矩阵乘法与二阶行列式。
- 辅助判断洛必达条件。
- 支持概率分布、密度函数归一化等后续扩展。

这种设计并不宣称“完全消灭幻觉”，而是对 MVP 覆盖范围内可符号化、可计算的关键步骤进行确定性校验，显著降低通用大模型在严谨推理任务中的错误风险。

### 2.5 错因诊断与学习闭环

系统不仅判断“对或错”，更关注“为什么错”。当前错因规则覆盖高数、线代、概率论中的常见错误，例如：

- 幂函数积分漏除以 `n+1`。
- 不定积分漏写常数 `+C`。
- 链式法则内外层混淆。
- 洛必达法则适用条件判断不清。
- 二阶行列式主副对角线相减顺序错误。
- 矩阵乘法维度或行列对应错误。
- 条件概率分母选错。
- 混淆互斥与独立。

学习闭环为：

```text
答题/提交步骤
  -> 工具验证
  -> 错因识别
  -> 生成 Tutor 式反馈
  -> 保存尝试与错因
  -> 更新掌握度
  -> 生成复盘卡片
  -> 推荐类似题或学习资源
```

这使项目从单次答疑工具升级为持续学习系统。

### 2.6 模型接入方案

模型层采用 OpenAI-compatible 统一接口：

```python
class LLMClient:
    async def stream(self, messages: list[dict], api_key: str | None = None):
        ...
```

当前默认对接 DeepSeek，并保留对其他 OpenAI-compatible 服务的扩展能力。配置项包括：

```env
LLM_PROVIDER=deepseek
LLM_BASE_URL=https://api.deepseek.com
LLM_API_KEY=
LLM_MODEL=deepseek-chat
ALLOW_USER_API_KEY=true
```

模型策略：

- 平台可配置统一 Key，适合演示与统一部署。
- 用户可在前端使用自己的 API Key，Key 仅保存在浏览器本地，不进入后端数据库。
- 请求级模型选择支持 `deepseek-v4-flash`、`deepseek-v4-pro`、`deepseek-chat`。
- 后端对模型名进行 allowlist 校验，非法模型回退到环境变量默认模型。

### 2.7 数据与隐私设计

MVP 阶段采用 SQLite 轻状态，不做复杂登录、长期画像或多设备同步。

保存内容：

- 用户默认标识：`demo-user`。
- 会话：课程、标题、创建和更新时间。
- 消息：用户与助手内容、意图、时间。
- 尝试：题目、学生步骤、验证结果、错因、验证摘要。
- 错因事件：学科、知识点、错因代码、时间。
- 掌握度：围绕概念的轻量分数与标签。

不保存内容：

- 用户 API Key。
- 复杂身份认证信息。
- 第三方账号隐私数据。
- 未授权教材全文再分发数据。

## 4. 创新点

### 4.1 可验证推理：从“会说”到“会验”

通用大模型擅长表达，但不擅长保证每一步计算可靠。珞珈数智助教将 LLM 的自然语言能力与 SymPy/Python 的确定性计算结合起来，使智能体在关键步骤上能够调用工具验证，而不是只靠语言流畅度取信用户。

### 4.2 启发式教学：从“给答案”到“搭脚手架”

项目内置引导模式、直接讲解模式和练习模式。默认推荐引导式反馈，通过分层提示帮助学生保留思考空间；当用户明确要求完整解答时，再切换为直接讲解。这比普通搜题工具更符合学习过程。

### 4.3 错因记忆：从“单次问答”到“持续追踪”

系统会记录错因事件、尝试结果和概念掌握度，使智能体能够知道学生不是笼统地“不会”，而是具体在“链式法则”“条件概率分母”“互斥独立区分”等点上反复出错。

### 4.4 自适应复练：从“讲完即止”到“诊断-练习-复盘”

系统可以围绕当前错因生成类似题，并在右侧学习面板中展示验证状态、错因、掌握度和下一步建议。这使产品形成完整学习闭环，而不是停留在答案生成。

### 4.5 课程级智能体模板：从数学样板到 STEM 平台

项目未来可将每门课程抽象为：

```text
Course Agent = Knowledge + Tools + Misconceptions + Exercises + Hint Policy + Analytics
```

这样大学数学中的成功经验可以迁移到物理、统计、电路等课程。扩展新课程时，重点不是重写整个产品，而是补充课程知识库、工具链、错因规则和练习模板。

## 5. 竞品分析

| 类型 | 代表产品 | 局限 | 珞珈数智助教优势 |
|---|---|---|---|
| 通用大模型 | ChatGPT、DeepSeek、文心一言等 | 数学计算可能幻觉，缺少稳定学习状态追踪 | 引入本地知识库、符号工具验证和错因记忆 |
| 搜题 APP | 作业帮、小猿搜题等 | 直接给答案，重题库匹配，难以批改学生自写步骤 | 支持自然语言步骤检查和启发式引导 |
| 科学计算工具 | WolframAlpha、Mathematica、Matlab | 门槛高，偏结果计算，缺少教学互动 | 自然语言交互，工具调用对学生隐形 |
| 普通 AI Tutor | 各类教育聊天机器人 | 多为文本问答，缺少确定性验证和错因闭环 | LLM + Knowledge + Verification + Memory + Practice Loop |

核心竞争力：

- 以强验证场景切入，技术主线清晰。
- 已有本地 Web App 和后端闭环，展示性强。
- 符号验证、错因记录、类似题生成共同构成学习闭环。
- 从大学数学可自然延伸到更多 STEM 课程，具备平台化空间。

## 6. 应用落地计划

### 6.1 近期：本地 MVP 与高校小范围试点

目标：把大学数学 Tutor 闭环做扎实。

重点任务：

- 补充高数、线代、概率论错因规则。

## 8. 风险与应对

| 风险 | 表现 | 应对策略 |
|---|---|---|
| 大模型计算幻觉 | 生成流畅但错误的推导 | 对可符号化步骤调用 SymPy/Python 验证 |
| 错因规则覆盖不足 | 新题型无法准确归因 | 从 MVP 题型开始，持续扩展规则与评测集 |
| 用户过度依赖答案 | 学生只复制完整解答 | 默认引导模式，完整解答需用户主动请求 |
| 教材与资源授权风险 | 不适合公开再分发部分资料 | 使用结构化自建知识片段，公开材料避免分发未授权全文 |
| 商业化叙事过早 | 数据不足支撑规模化宣传 | 区分“已实现、试点、规划”，先做真实小样本评测 |
| 多课程扩展过散 | 过早做全学科导致失焦 | 先用数学打磨闭环，再选 1-2 门 STEM 课程复制 |
