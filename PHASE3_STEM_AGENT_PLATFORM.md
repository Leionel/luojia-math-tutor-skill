# 第三阶段规划：STEM 可验证学习智能体平台

## 1. 阶段定位

第三阶段不再只是“把数学助教做得更完整”，而是把珞珈数智助教升级为：

> 面向 STEM 教育的可验证学习智能体平台。

大学数学仍然是首个样板场景，但项目叙事要从“数学题答疑”上升到“严谨推理类课程的智能体基础设施”。

第三阶段的核心问题是：

> 如何把已经在大学数学中验证过的能力，迁移到更多 STEM 课程，并服务学生、教师和教育机构？

## 2. 第三阶段可以做什么

### 2.1 多课程扩展

从高数、线代、概率论扩展到更多需要严谨推理和工具验证的课程。

优先级建议：

1. 大学物理
   - 公式推导。
   - 量纲检查。
   - 运动学、力学、电磁学基础计算。
   - 数值仿真与曲线绘制。

2. 电路与信号系统
   - 节点方程。
   - 矩阵求解。
   - 频响曲线。
   - 拉普拉斯变换基础。

3. 统计与数据分析
   - 假设检验。
   - 置信区间。
   - 回归分析。
   - 分布计算与可视化。

4. 控制与优化基础
   - 传递函数。
   - 稳定性判断。
   - 梯度下降。
   - 简单约束优化。

省赛表达建议：

> 以大学数学为起点，逐步沉淀“课程知识库 + 错因库 + 工具链 + 练习库 + 教学策略”的课程级智能体模板，面向 STEM 基础课复制。

### 2.2 教师端学习分析面板

学生端解决“我不会怎么办”，教师端解决“全班哪里薄弱”。

教师端可以做：

- 班级错因热力图。
- 知识点薄弱排名。
- 作业步骤错误统计。
- 高频错因样例聚类。
- 针对薄弱知识点自动生成补充练习。
- 期末复习重点建议。

不建议一开始做完整教师后台，可以先做一个最小看板：

```text
课程/班级
-> 知识点掌握度分布
-> Top 错因
-> 高频问题
-> 推荐复习任务
```

省赛表达建议：

> 项目不仅服务学生个体，也能为教师提供班级级别的学习诊断，辅助公共基础课教学提质增效。

### 2.3 可验证工具链平台化

第三阶段要把 SymPy 验算从“数学内部工具”升级为“工具调用层”。

可以抽象为：

```text
Tool Registry
-> Symbolic Math Tool
-> Numeric Simulation Tool
-> Plotting Tool
-> Unit/Dimension Check Tool
-> Statistical Test Tool
```

每门课配置不同工具：

- 数学：SymPy、矩阵计算、极限/积分/求导。
- 物理：量纲检查、运动方程仿真。
- 电路：矩阵方程、频响计算。
- 统计：分布函数、假设检验、回归。

省赛表达建议：

> 智能体不是单纯调用大模型，而是通过工具注册与调度机制，把生成式 AI 的表达能力与确定性计算工具结合起来。

### 2.4 多模态可视化学习组件

第三阶段可强化“概念可视化”，但它应服务学习闭环，而不是单独炫技。

可视化方向：

- 函数曲线。
- 积分面积/体积。
- 梯度向量场。
- 概率密度曲线。
- 矩阵线性变换动画。
- 物理运动轨迹。
- 电路频响曲线。

产品表达：

> 智能体根据概念类型自动选择表达形式：文字解释、公式推导、图像可视化、交互练习。

MVP 做法：

- 后端生成图片或数据点。
- 前端在右侧学习面板中展示。
- 聊天区引用可视化结果解释。

### 2.5 课程级智能体模板

第三阶段最值得做成平台能力的是“课程模板”。

每门课都包含：

- 课程知识库。
- 典型题型库。
- 错因规则库。
- 工具调用配置。
- 提示策略。
- 练习生成模板。
- 教师看板指标。

抽象结构：

```text
Course Agent = Knowledge + Tools + Misconceptions + Exercises + Hint Policy + Analytics
```

这样项目就从一个单独应用变成可复制平台。

省赛表达建议：

> 我们不是为每门课重新开发一个 AI 助教，而是沉淀一套课程级智能体生成框架，使不同 STEM 课程可以通过配置知识库、工具链和错因库快速接入。

### 2.6 B 端与 API 服务

第三阶段可以规划 B 端接口，但不要夸张成已经商业化。

可服务对象：

- 高校公共基础课。
- 在线教育平台。
- 教培机构。
- 学习工具 App。
- 教师教学辅助系统。

可提供能力：

- Tutor Agent API。
- Step Checker API。
- Mistake Diagnosis API。
- Similar Exercise API。
- Learning Analytics API。

商业路径：

```text
学生端免费/低价工具
-> 高校课程试点
-> 教师端看板
-> 教培机构/在线教育平台 API
-> 课程级智能体解决方案
```

## 3. 第三阶段不建议做什么

为了避免摊子过大，第三阶段也要有边界。

不建议优先做：

- 自研大模型。
- 大规模模型训练。
- 全学科一次性覆盖。
- 完整教务系统。
- 重型 LMS 平台。
- 复杂登录和多租户权限体系。
- 没有验证数据支撑的“99.9% 准确率”宣传。

更稳的策略是：

> 先用大学数学证明方法有效，再复制到 2-3 门 STEM 课程，最后用教师端和 API 证明产业价值。

## 4. 第三阶段路线图

### 4.1 近期：1-2 个月

目标：把数学学习闭环做扎实。

- 完善 mastery。
- 完善错因复盘卡片。
- 完善类似题生成。
- 增加 20-50 道验收题。
- 记录验算命中率、错因识别率、复练正确率。

### 4.2 中期：3-6 个月

目标：扩展到 STEM 课程样板。

- 选择大学物理或统计作为第二课程。
- 建立课程知识库。
- 补充对应工具链。
- 增加课程错因规则。
- 做跨课程 Demo。

### 4.3 长期：6-12 个月

目标：形成平台化能力。

- 教师端学习分析面板。
- 课程级智能体模板。
- API 服务。
- 高校课程试点。
- 教培机构合作验证。

## 5. PPT 中如何讲第三阶段

第三阶段不要讲太散，建议放在“未来规划”一页中。

可以写成三层：

```text
第一层：大学数学 Tutor Agent
已实现：对话、检索、验算、错因、类似题。

第二层：课程级学习闭环
推进中：掌握度、复练路径、教师看板。

第三层：STEM 可验证学习智能体平台
未来：物理/电路/统计等课程接入，提供教育智能体 API。
```

一句话总结：

> 从“一个会教数学的智能体”，升级为“严谨推理类课程的可验证智能体平台”。

## 6. 计划书中如何写第三阶段

可以放在“未来规划与商业化路径”部分：

> 项目第三阶段将围绕 STEM 教育场景扩展，沉淀课程级智能体模板。每门课程由知识库、错因库、工具链、练习库和教学策略组成。平台将优先扩展大学物理、统计与电路等强推理课程，并面向高校公共基础课、在线教育平台和教培机构提供智能答疑、步骤诊断、错因分析和学习数据看板能力。

## 7. 第三阶段核心卖点

最重要的卖点不是“学科更多”，而是：

1. 可复制。
   - 一套智能体框架可以接入多门课程。

2. 可验证。
   - 对严谨推理步骤引入确定性工具检查。

3. 可持续。
   - 有错因、掌握度和复练路径。

4. 可落地。
   - 学生端、教师端、机构端都有应用路径。

5. 可商业化。
   - 未来可以提供课程智能体 API 和教学分析服务。

## 8. 最终建议

第三阶段可以做，但不要说成“我们要做所有教育场景”。更合适的表达是：

> 以大学数学为首个验证场景，逐步扩展到物理、统计、电路等 STEM 基础课程，形成可验证、可配置、可复用的课程级学习智能体平台。

这条路线既有技术厚度，也有产业想象空间，还不会显得空泛。
