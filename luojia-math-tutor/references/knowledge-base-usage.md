# 珞珈数智助教本地知识库使用指南

本指南说明如何检索并使用 `references/` 下的结构化知识库与教材资料。目标是让回答尽量有来源、有定义、有边界，而不是凭印象讲题。

## 1. 什么时候读取本指南

在以下场景优先读取并执行本指南：

* 用户问“什么是/为什么/怎么理解”某个数学概念。
* 解题前需要确认定理、定义、公式或适用条件。
* 用户要求“按教材讲”“给权威定义”“考研/期末复习”。
* 模型对某个符号、定理名或章节归属不确定。

若只是非常基础的四则运算或用户明确只要简短答案，可以不检索知识库，但不要编造教材出处。

## 2. 结构化知识库

路径：`references/output/`

| 文件 | 主要内容 | 优先使用场景 |
| :--- | :--- | :--- |
| `concept_gs.json` | 高等数学、微积分、函数、极限、导数、积分、级数等 | 高数概念讲解与基础推导 |
| `la.json` | 行列式、矩阵、线性方程组、特征值、二次型等 | 线性代数讲题、定义核查 |
| `proba.json` | 概率论与数理统计核心定义、定理、公式 | 概率论概念与公式检索 |
| `Proba_example.json` | 概率论、数理统计、随机过程相关例题与扩展条目 | 概率例题、应用题、随机过程问题 |
| `ex.json` | 微积分习题与示例 | 需要参考相似例题时 |

常见字段：

* `concept_zh`：概念、定理、公式或例题名称。
* `prerequisite`：前置知识，用来判断学生可能卡在哪里。
* `description` / `core_content_latex`：标准定义、定理表述或题目内容。
* `intuitive_explanation`：直观解释，可作为类比讲解的素材。
* `solution`：例题解法或补充说明。使用时要改写成启发式讲解，不要整段照搬。

## 3. 检索流程

1. 先抽取关键词：包括中文概念名、常见别名、英文名和关键符号。例如“特征值/特征向量/eigenvalue/`Ax=λx`”。
2. 先查最匹配的 JSON 文件；不确定学科时再跨文件检索。
3. 在命令行环境优先使用全文搜索，例如 `rg "二阶行列式" references/output/la.json`。在其他 AI 客户端中使用等价的文件搜索或文件读取工具。
4. 找到候选条目后，核对 `concept_zh`、`prerequisite` 和正文，不要只凭关键词命中。
5. 回答时融合条目内容，避免机械粘贴；必要时指出“这里用到的前提是……”。

检索不到时：

* 尝试同义词、英文名、符号形式和上位概念。
* 若仍无结果，说明“本地结构化库未命中该概念”，再用通用数学知识讲解。
* 不要伪造“库中写道”“教材第几页”之类来源。

## 4. 教材 PDF 库

路径：`references/textbook/`

PDF 只作为兜底材料使用，优先级低于结构化 JSON。适合以下情况：

* JSON 内容不足以处理冷门概念或高阶课程内容。
* 用户要求追溯教材表述、章节上下文或更严格证明。
* 需要确认某个定义在不同教材中的表述差异。

使用原则：

1. 先根据文件名判断可能相关教材。
2. 若当前环境支持 PDF 文本提取，可以检索目录、章节名或关键词。
3. 若 PDF 无法读取或疑似扫描件，不要硬编页码；可以建议用户提供截图或指定章节。
4. 对教材内容做摘要和教学化改写，不要大段复刻原文。

## 5. 输出时如何引用知识库

对学生输出时不必暴露内部文件路径，除非用户要求溯源。推荐说法：

* “按本地知识库里的标准表述，这个概念核心是……”
* “这一步的前提是……，如果不满足就不能直接用。”
* “我先用结构化库核了一下定义，再把它换成更好理解的说法。”

回答结构建议：

1. **知识点定位**：指出命中的概念或定理。
2. **前提条件**：列出使用该公式/定理前必须满足的条件。
3. **直观解释**：用生活化或几何化方式翻译。
4. **当前题目连接**：说明这个知识点在题目里负责哪一步。

## 6. 质量底线

* 不把检索结果当成绝对正确；遇到公式、符号或条件异常时，用计算工具或教材复核。
* 不把例题答案直接当作用户题目的答案；先比较题型和条件。
* 不为展示“权威感”而引用未实际读取的资料。
* 对概率统计、数值计算、矩阵运算等容易出错的内容，结合 `math-tools-guidelines.md` 做后台验算。
